The Business Case: ROI, Scoping & Change Management

Technical skill alone doesn't ship an agent. Someone has to decide it's worth building, scope it correctly, and manage what changes for the people whose work it touches.

This module is written for anyone who needs to greenlight, scope, or manage the rollout of an agentic project -- not just the people writing the code.

Scoping an agent project realistically

  • Start with a bounded, well-understood task, not "automate customer support." A good first project has a clear success metric and a human fallback that already exists.
  • Estimate cost realistically: per-session token cost x expected session volume x expected steps per session, using the worst-case step count from Module 12, not just the best case seen in a demo.
  • Identify the human role that changes, not just the one that's "replaced." Most successful agent deployments shift a human from doing a task to reviewing or handling the exceptions the agent escalates -- plan for that new role explicitly.
  • Define what "good enough" means before launch, using the eval set from Module 10 -- not "let's see how it feels once it's live."
The most common failure isn't technical

Many agentic AI projects don't fail because the agent doesn't work -- they fail because nobody defined what success looked like, nobody planned for the exceptions the agent would surface, or the people whose workflow changed weren't brought in early enough to trust it.

For managers

Treat an agent rollout as a change management project with a technical component, not a technical project with a rollout attached. The team who will live with the agent day to day should help define its escalation rules (Module 11), not just receive them.

Even non-engineers benefit from being able to read a rough cost model, since it's usually the deciding factor in whether a project is viable at scale.

# A back-of-envelope cost model for a support-triage agent
sessions_per_month = 20_000
avg_steps_per_session = 4          # typical, not worst case
avg_tokens_per_step = 1_500        # input + output combined
cost_per_1k_tokens = 0.01          # illustrative, check current model pricing

monthly_cost = (
    sessions_per_month * avg_steps_per_session * avg_tokens_per_step / 1000
) * cost_per_1k_tokens

# Always also compute the worst case: what if every session hits
# the max_steps cap from Module 12? That's your real budget ceiling.
worst_case_cost = (
    sessions_per_month * MAX_STEPS * avg_tokens_per_step / 1000
) * cost_per_1k_tokens

Present both numbers to stakeholders, not just the typical case -- the gap between them is exactly the risk a step-cap and good prompt design (Modules 3 and 12) are meant to control.

A simple scoping checklist for a first agent project

  • Is there a clear, measurable definition of success?
  • Is there an existing human fallback for when the agent can't handle something?
  • Have the people doing this work today been involved in defining the escalation rules?
  • Is there a realistic cost estimate, including the worst case?
Exercise 13.1 -- Scope a first project

Apply the scoping checklist to a real proposal

A colleague proposes: "Let's build an agent that handles all inbound customer emails end to end." Using the checklist above, identify what's missing from this proposal and rewrite it as a better-scoped first project.

Show one possible answer

What's missing: no bounded scope ("all" emails, not a specific category), no stated success metric, no mention of a human fallback, no cost estimate. A better-scoped version: "Build an agent that drafts responses to password-reset and billing-question emails specifically (our two highest-volume, most templated categories), with every draft reviewed by a human before sending for the first month, measured by percentage of drafts sent without edits."
