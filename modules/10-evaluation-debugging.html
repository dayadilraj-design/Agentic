Evaluation, Testing & Debugging Agents

Agents fail in ways traditional software doesn't -- silently, plausibly, and inconsistently across runs. Without a real evaluation practice, you're shipping on vibes.

Because the same agent can behave differently on the same input (models aren't fully deterministic, and small context changes cascade), you can't rely on the traditional unit-test mindset of "run once, assert pass." You need evaluation as an ongoing practice, not a one-time gate.

Three layers of agent evaluation

  • Component-level tests -- does each individual tool work correctly, deterministically, in isolation? This is normal software testing and it's non-negotiable.
  • Trajectory evaluation -- given a task, did the agent take a reasonable path to the answer (right tools, right order), not just a correct final output? A right answer reached the wrong way is a ticking time bomb.
  • Outcome evaluation at scale -- running a representative set of test cases (an "eval set") repeatedly, tracking success rate over time as you change prompts, models, or tools.
Build your eval set early

Collect real, representative examples of tasks your agent needs to handle -- including edge cases and known-hard inputs -- before you need them. An eval set built after a production incident is a eval set built too late.

For managers

Ask "what's the eval set, and what's the current pass rate?" the same way you'd ask about test coverage on any other software project. If the answer is "we tried it a few times and it seemed fine," treat that as a yellow flag, not a green light.

A simple but genuinely useful evaluation harness: run the agent against a fixed set of test cases, and score each on both outcome and trajectory.

eval_cases = [
    {"input": "Book a flight from Delhi to Mumbai next Friday",
     "expected_tool_sequence": ["search_flights", "confirm_with_user", "book_flight"],
     "expected_outcome_contains": "confirmation number"},
    # ... more cases covering edge cases: no flights found, ambiguous dates, etc.
]

def run_eval(agent, cases):
    results = []
    for case in cases:
        trace = agent.run(case["input"], return_trace=True)
        tool_sequence_ok = trace.tool_names == case["expected_tool_sequence"]
        outcome_ok = case["expected_outcome_contains"] in trace.final_output
        results.append({
            "case": case["input"],
            "trajectory_correct": tool_sequence_ok,
            "outcome_correct": outcome_ok
        })
    return results

# Run this eval set every time you change the prompt, model version,
# or tool definitions -- treat it exactly like a regression test suite.

Using an LLM as a judge

For outputs too open-ended to check with exact string matching (like the quality of a written summary), a common pattern is to use a second model call with a clear rubric to score the output -- while being aware this "LLM-as-judge" approach has its own biases and needs occasional human spot-checking to stay trustworthy.

Write three test cases for a scheduling agent

For an agent that books meetings, write three eval cases: one straightforward booking, one where the requested time conflicts with an existing meeting, and one where the user's request is ambiguous (e.g. "sometime next week"). For each, specify the expected tool sequence and a way to check the outcome.

Show one possible answer

Case 1 (straightforward): expected tools: check_availability then book_meeting; outcome check: response contains a confirmation.
Case 2 (conflict): expected tools: check_availability then a response offering alternate times, not book_meeting; outcome check: response mentions the conflict and proposes alternatives.
Case 3 (ambiguous): expected behavior: agent asks a clarifying question rather than guessing a specific time; outcome check: response contains a question, and no booking tool was called.
