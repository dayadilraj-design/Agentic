Retrieval-Augmented Generation (RAG) for Agents

Agents that reason over your company's actual documents -- not just the model's training data -- need RAG. It's the single most common way to ground an agent in real, current, private information.

RAG answers a simple problem: your documents aren't in the model's training data, and you can't fit your entire knowledge base into every prompt. The fix is to search for the relevant few passages at query time, and only inject those.

The RAG pipeline, end to end

  • Chunk -- split documents into passages small enough to retrieve individually (typically a few hundred words).
  • Embed -- convert each chunk into a vector that captures its meaning.
  • Store -- index those vectors in a vector database for fast similarity search.
  • Retrieve -- at query time, embed the user's question and fetch the most similar chunks.
  • Generate -- feed those chunks into the model's context alongside the question, and ask it to answer grounded only in what was retrieved.
RAG in an agent vs. a chatbot

In a simple chatbot, RAG usually happens once per turn. In an agent, retrieval becomes a tool the model can call repeatedly and adaptively -- searching again with a refined query if the first results weren't good enough, which is a meaningfully more powerful pattern.

For managers

"Hallucination" complaints about an internal AI tool are very often a retrieval problem, not a model problem -- the right document existed but wasn't found. Before blaming the model, check what was actually retrieved for that query.

Here's a RAG-as-a-tool setup, which lets an agent decide when and how many times to search.

# Define retrieval as a callable tool, just like any other
tools = [{
    "name": "search_docs",
    "description": "Search internal documentation for relevant passages",
    "input_schema": {
        "type": "object",
        "properties": {"query": {"type": "string"}},
        "required": ["query"]
    }
}]

def search_docs(query):
    query_vector = embed_model.encode(query)
    results = vector_store.query(vector=query_vector, top_k=5)
    return [r.metadata["text"] for r in results]

# The agent loop treats this exactly like the weather tool from Module 4 --
# the model decides when to call it, and can call it again with a
# refined query if the first results don't fully answer the question.

Chunking strategy matters more than people expect

Chunks that are too large dilute relevance (the right sentence gets buried among irrelevant ones); chunks that are too small lose context (a fact separated from the heading that explains it). A common starting point is 200-500 word chunks with modest overlap between them, then tuning based on retrieval quality on real queries.

Exercise 7.1 -- Debug a bad RAG answer

Diagnose why an agent gave a wrong answer from your own docs

An internal HR agent, asked "how many vacation days do senior engineers get," answers incorrectly even though the correct number is in your handbook. Walk through the pipeline (chunk / embed / store / retrieve / generate) and list one concrete thing to check at each stage.

Show one possible answer

Chunk: is the vacation policy split across two chunks, separating "senior engineer" from the actual number? Embed: does the embedding model handle role-specific terms like "senior engineer" well, or is it too generic? Store: is the handbook actually indexed, or is it an older version? Retrieve: print the top_k chunks for this exact query -- is the right passage even coming back? Generate: if the right chunk was retrieved but the answer is still wrong, the issue is prompting (the model may be ignoring the retrieved context), not retrieval.
