A Tour of Agent Frameworks

You now understand the loop well enough to build it by hand -- which means you're also ready to judge which framework (if any) is worth adopting, instead of cargo-culting whatever's trending.

Every framework in this space is implementing some version of the Perceive-Reason-Act-Reflect loop from Module 1. What differs is how much structure they impose, and what they optimize for.

The landscape, at a glance

  • Direct API + tool use (what you've been doing all course) -- maximum control, minimum abstraction. Best when you need precise, custom behavior or are building something novel.
  • LangChain / LangGraph -- a large ecosystem of pre-built components plus a graph-based orchestration layer (LangGraph) for defining agent state machines explicitly. Strong when you want fine control over branching logic and many integrations.
  • CrewAI -- built around the metaphor of a "crew" of role-based agents (researcher, writer, reviewer) collaborating on a task. Strong for quickly prototyping multi-agent workflows with minimal boilerplate.
  • AutoGen -- a conversation-centric framework where agents "talk" to each other and to humans in a shared chat thread. Strong for research-style, exploratory multi-agent setups.
  • Claude Agent SDK -- Anthropic's own SDK for building agents directly against Claude, with first-class support for tool use, computer use, and long-running agent loops.
The uncomfortable truth

Frameworks add real value for standardizing common patterns and speeding up prototyping -- but every one of them can also become a debugging tax once something goes wrong three abstraction layers deep. Many production teams start with a framework and later strip it back to a thin custom loop once they understand their exact needs.

For managers

Framework choice is a real but reversible decision -- treat it like picking a web framework, not like picking a database. Weight developer familiarity and community support alongside features; the "best" framework on paper is often not the one your team can debug at 2am.

To make the comparison concrete, here's the same tiny task -- "research a topic, then write a two-sentence summary" -- sketched in a raw loop versus a framework-assisted style.

# Raw loop (what you built in Modules 1-7)
result = agent_loop(
    goal="Research topic X and summarize in 2 sentences",
    tools=[search_tool, summarize_tool],
    max_steps=5
)

# Graph-style orchestration (LangGraph-flavored pseudocode)
graph = StateGraph()
graph.add_node("research", research_step)
graph.add_node("summarize", summarize_step)
graph.add_edge("research", "summarize")
graph.set_entry_point("research")
result = graph.run(initial_state={"topic": "X"})

# Role-based crew (CrewAI-flavored pseudocode)
researcher = Agent(role="Researcher", tools=[search_tool])
writer = Agent(role="Writer", tools=[summarize_tool])
crew = Crew(agents=[researcher, writer], tasks=[research_task, write_task])
result = crew.kickoff()

How to actually choose

  • Start with the raw loop for anything novel or safety-sensitive -- you need to understand exactly what's happening.
  • Reach for a graph-based framework once your control flow has real branches and loops that get hard to track by hand.
  • Reach for a role-based framework when the natural decomposition of the task really is "specialist roles collaborating," not just a sequence of function calls.
Exercise 8.1 -- Pick a framework and justify it

Choose the right tool for three different projects

For each project, pick raw-loop, graph-based, or role-based, and justify in one sentence:

  1. A financial approval agent where every decision must be auditable step-by-step and mistakes are costly.
  2. A content pipeline where a "researcher," "writer," and "editor" each do a distinct pass on a blog post.
  3. A prototype you're building in an afternoon to see if agentic search is even worth pursuing for your product.
Show suggested answers

1. Raw loop -- you want full visibility and control over every step when mistakes are costly and auditability matters most.
2. Role-based (e.g. CrewAI-style) -- the task naturally maps to distinct specialist roles collaborating in sequence.
3. Whichever framework your team already knows best -- for a same-day prototype, familiarity beats theoretical fit.
