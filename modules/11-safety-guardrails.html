Safety, Guardrails & Human-in-the-Loop

The gap between a clever demo and a production agent is almost entirely made of guardrails. This module is the difference between an agent you can trust with real consequences and one you can't.

Every capability you give an agent is also a risk surface. Safety in agentic systems isn't a single feature you bolt on -- it's a set of layered decisions about what the agent can do, what it must confirm first, and what it should never do at all.

A layered approach to agent safety

  • Least privilege -- give each agent only the tools and data access it strictly needs for its task, nothing broader "just in case."
  • Tiered autonomy -- classify actions by risk: fully autonomous (read a file), confirm-first (send an external email), always-human (anything involving money, legal exposure, or irreversible deletion).
  • Input and output validation -- sanitize what goes into the agent (especially from untrusted sources like scraped web content) and validate what comes out before it triggers a real action.
  • Rate and scope limits -- cap how many actions, how much spend, or how many API calls an agent can make in a session, so a bug or bad decision fails small instead of large.
  • Auditability -- log every decision and tool call with enough detail to reconstruct exactly what happened and why, after the fact.
The prompt injection risk

An agent that reads untrusted content (a webpage, an email, a document from an external party) can have instructions embedded in that content mistaken for legitimate commands -- this is called prompt injection. Never let content an agent merely reads carry the same authority as an instruction from its actual operator; treat all fetched content as data, not commands.

For managers

Before approving any agent with real-world side effects, ask specifically: what's the worst single action this agent could take, and what stops it from taking that action unsupervised? If there's no concrete answer, that's the first gap to close -- not a documentation exercise to do later.

Here's a tiered-autonomy pattern that gates risky tool calls behind human confirmation, without blocking low-risk ones.

TOOL_RISK_TIERS = {
    "search_docs": "autonomous",
    "read_file": "autonomous",
    "send_email": "confirm_first",
    "delete_record": "always_human",
    "issue_refund": "always_human",
}

def execute_tool_call(tool_name, args, session):
    tier = TOOL_RISK_TIERS.get(tool_name, "always_human")  # unknown tools default to strictest tier

    if tier == "autonomous":
        return run_tool(tool_name, args)

    if tier == "confirm_first":
        if session.get_user_confirmation(f"Agent wants to {tool_name} with {args}. Proceed?"):
            return run_tool(tool_name, args)
        return {"status": "declined_by_user"}

    # always_human tier: never auto-executed under any circumstance
    return {"status": "escalated", "message": "This action requires human execution"}

Defending against prompt injection

Practical mitigations include: clearly delimiting untrusted content in the prompt (so the model can distinguish "data to read" from "instructions to follow"), stripping or flagging suspicious instruction-like text in fetched content, and never granting an agent that reads untrusted external content the same tool access as one that only handles trusted, internal input.

Exercise 11.1 -- Tier a real tool list

Classify six tools by risk tier

Assign each of the following tools to autonomous / confirm_first / always_human, and justify any that feel like a close call: search_web, post_to_company_twitter, read_customer_record, update_customer_address, transfer_funds, generate_draft_reply.

Show one possible answer

search_web -- autonomous (read-only, low risk). read_customer_record -- autonomous, assuming proper access controls are already in place at the data layer. generate_draft_reply -- autonomous (drafting isn't sending). update_customer_address -- confirm_first (reversible but affects real records). post_to_company_twitter -- confirm_first (public-facing, reputational risk, but reversible by deleting). transfer_funds -- always_human (financial, often irreversible, highest stakes).
