Prompting Patterns for Agents

System prompts for agents do more work than chatbot prompts ever did -- they define identity, boundaries, available tools, and how to handle uncertainty. Get this wrong and everything downstream inherits the problem.

A chatbot's system prompt sets tone. An agent's system prompt sets behavior under ambiguity -- what it should do when a tool fails, when instructions conflict, or when it is not sure it has enough information.

What an agent's system prompt needs to cover

  • Role and scope -- what the agent is for, and just as importantly, what it is not for.
  • Available tools -- what each tool does, when to use it, and when not to.
  • Stopping conditions -- how the agent knows it is done, so it doesn't loop forever or stop too early.
  • Failure handling -- what to do if a tool call errors, returns nothing, or returns something unexpected.
  • Escalation rules -- when to ask a human instead of guessing or acting.
Pattern: ReAct

ReAct (Reason + Act) is the most common agent prompting pattern: the model alternates between writing out a reasoning step and taking an action, observing the result before reasoning again. It is simple, transparent, and forms the backbone of most agent frameworks you'll meet in Module 8.

For managers

A surprising amount of "AI agent" project risk lives in the system prompt, not the model. Ask to see it. If it doesn't specify what happens on failure, that's a gap worth closing before launch.

Here is a realistic system prompt skeleton for a support-ticket triage agent, annotated to show what each section is doing.

system_prompt = """
You are a support ticket triage agent for Acme Software.

# ROLE AND SCOPE
Your job is to read incoming tickets, categorize them, and either
resolve simple ones directly or route complex ones to a human.
You do not have access to billing systems or the ability to issue refunds.

# AVAILABLE TOOLS
- search_kb(query): search the internal knowledge base
- create_ticket(category, priority, summary): file a ticket for a human
- reply_to_customer(message): send a message back to the customer

# STOPPING CONDITIONS
Stop and reply once you have either fully resolved the issue using
the knowledge base, or filed a ticket with enough detail for a human
to pick it up without re-asking the customer basic questions.

# FAILURE HANDLING
If search_kb returns no relevant results after 2 attempts with
different queries, file a ticket rather than guessing an answer.

# ESCALATION
Any mention of legal threats, safety issues, or refund requests over
$500 must be filed as priority "urgent" and never answered directly.
"""

Notice this prompt spends most of its words on edge cases, not the happy path -- that is deliberate. The happy path (customer asks a simple question, KB has the answer) needs almost no guidance. The value of a well-written agent prompt is entirely in how it behaves when things are not simple.

Few-shot examples still matter

For agents with nuanced judgment calls (like the $500 refund threshold above), showing 2-3 worked examples of correct tool choices in ambiguous cases outperforms describing the rule in the abstract.

Exercise 3.1 -- Write the missing sections

Harden a thin system prompt

You are given this system prompt for a travel-booking agent:

"You are a travel booking assistant. You can search flights and book them for the user."

Rewrite it to add: (1) a spending limit escalation rule, (2) a failure-handling instruction for when the flight search API times out, and (3) a stopping condition. Aim for 6-10 sentences total.

Show one possible answer

"You are a travel booking assistant. You can search flights using search_flights() and book them using book_flight(). Never book a flight over $800 without first presenting options and getting explicit user confirmation of the exact flight and price. If search_flights() times out or errors, retry once; if it fails again, tell the user you're unable to search right now rather than guessing at flight availability. Consider the task complete once you have either booked a confirmed flight and shown the confirmation number, or determined that no suitable flight exists and explained why."

Knowledge check

1. What does the ReAct pattern alternate between?

2. Why do most of the words in a good agent system prompt address edge cases rather than the happy path?

3. A support agent has no tool for issuing refunds. What should its system prompt do about refund requests?
