What Is an AI Agent, Really?

Before you build anything, you need a definition that survives contact with a whiteboard argument. This module gives you one, and the vocabulary the rest of the course depends on.

A chatbot answers. An agent acts. That one-line distinction is the whole course — everything else is detail.

A large language model, on its own, is a very good next-word predictor. Ask it a question in a chat window and it replies with text. Nothing happens in the world. An agent wraps that same model in a loop that lets it observe a situation, decide what to do, actually do something (call a tool, run code, query a database, send an email), look at the result, and decide again. That loop — Perceive → Reason → Act → Reflect — is the spine of this entire course, and you'll see it tagged on every module.

The four-part loop

  • Perceive — the agent gathers information: a user message, a document, an API response, the result of its last action.
  • Reason — the model decides what the information means and what to do about it, often thinking step by step.
  • Act — the agent takes a concrete action in the world: calling a tool, writing a file, hitting an API.
  • Reflect — the agent (or a supervising system) checks whether the action worked, and whether to continue, retry, or stop.

Almost every framework, pattern, and failure mode you will meet later is a variation on how tightly or loosely that loop is implemented.

Key distinction

"Autonomous" doesn't mean "unsupervised." A well-designed agent has guardrails, approval points, and limits. Autonomy is about who decides the next step, not about removing human oversight entirely.

For managers

When someone pitches you "an AI agent," ask three questions: what does it perceive, what actions can it actually take, and what happens when it's wrong. If they can't answer the third one, it's not ready for production.

Let's make the loop concrete with the smallest possible agent: one that can check the weather and decide whether to recommend an umbrella.

Even without any framework, an agent is just a loop around a model call, with a way for the model to request actions and a way for your code to run them:

# pseudocode -- the agent loop, stripped to its essence
state = {"goal": "Tell the user if they need an umbrella today"}

while not state["done"]:
    # 1. PERCEIVE -- give the model the current state
    response = model.call(system_prompt, state["history"])

    # 2. REASON -- the model decides: answer, or use a tool?
    if response.wants_tool:
        # 3. ACT -- your code executes the tool, not the model
        result = run_tool(response.tool_name, response.tool_args)
        state["history"].append(result)
    else:
        # 4. REFLECT -- model judged it has enough info to stop
        state["done"] = True
        final_answer = response.text

Notice what the model never does: it never calls the weather API directly. It only ever requests that a tool be called, in a structured format. Your code is the one thing standing between "the model wants to" and "it actually happened" -- which is exactly where your safety and business logic lives. We will build a real version of this loop with actual tool-calling in Module 4.

Agents vs. workflows vs. chatbots

These three get conflated constantly, so it is worth being precise:

  • Chatbot -- perceives and reasons, but does not act. Output is text back to a human.
  • Workflow / pipeline -- acts, but the sequence of actions is fixed by a human in advance.
  • Agent -- acts, and the model itself decides the sequence and which tools to call, based on what it perceives at each step.
Exercise 1.1 -- Classify the system

Sort five real systems into chatbot / workflow / agent

For each system below, decide which category it belongs to and write one sentence justifying your answer:

  1. A customer support widget that answers FAQ questions from a fixed knowledge base.
  2. A nightly script that pulls sales data, generates a PDF, and emails it to the sales team.
  3. A coding assistant that reads an error message, decides to search the codebase, edits a file, reruns the tests, and repeats until they pass.
  4. A form that routes a support ticket to "Billing" or "Technical" based on keyword matching.
  5. A research tool that, given a question, decides on its own whether to search the web, read a PDF, or ask you a clarifying question -- in whatever order it judges best.
Show suggested answers

1. Chatbot -- perceives and reasons, no ability to act on anything outside the conversation.
2. Workflow -- acts, but the sequence is fixed by a human, not decided by a model at runtime.
3. Agent -- the model decides the next action based on the result of the last one; the loop is genuinely closed.
4. Workflow (arguably not even AI) -- a fixed decision rule, not a reasoning loop.
5. Agent -- the ordering and choice of tools is decided dynamically by the model itself.
