Deploying Agents to Production

A working agent in a notebook and a working agent in production are two different engineering problems. This module covers the gap: cost, latency, monitoring, and graceful degradation.

Production agents need to handle the same concerns as any other production service -- uptime, cost control, observability -- plus a few that are specific to LLM-driven systems, like unpredictable step counts and non-deterministic behavior.

Production concerns specific to agents

  • Cost control -- an agent that loops indefinitely on a hard task can burn far more tokens than expected; set hard caps on steps, tokens, and tool calls per session.
  • Latency -- multi-step agent tasks are inherently slower than a single model call; design the UX around this (streaming partial progress, "agent is working on step 2 of 4") rather than a single long spinner.
  • Graceful degradation -- decide what happens when a tool is down, the model API is rate-limited, or the agent hits its step cap without finishing -- these should produce a clear, honest message, never a silent failure or a fabricated success.
  • Observability -- log full reasoning traces and tool calls (subject to your privacy policy), not just final outputs, so incidents are debuggable after the fact.
  • Versioning -- treat prompts, tool schemas, and model versions as versioned artifacts with rollback capability, exactly like application code.
Rollout strategy

Ship agent changes the way you'd ship any other risky change: behind a feature flag, to a small percentage of traffic first, with your eval set (Module 10) run as an automated gate before wider rollout.

For managers

Ask for a per-session cost estimate and a worst-case cost estimate (what happens if the agent hits max steps on every request) before approving a launch -- these numbers are often very different, and only one of them shows up in a demo.

A production-shaped agent loop adds hard limits and structured logging around the core loop from Module 1.

MAX_STEPS = 10
MAX_TOKENS_PER_SESSION = 50_000

def run_agent(goal, session_id):
    steps_taken = 0
    tokens_used = 0
    trace = []

    while steps_taken < MAX_STEPS and tokens_used < MAX_TOKENS_PER_SESSION:
        response = model.call(system_prompt, history)
        tokens_used += response.usage.total_tokens
        trace.append({"step": steps_taken, "action": response.summary()})  # observability

        if response.is_final:
            log_success(session_id, trace)
            return response.text

        try:
            result = run_tool(response.tool_name, response.tool_args)
        except ToolError as e:
            trace.append({"step": steps_taken, "error": str(e)})
            # feed the error back so the model can adapt, rather than crashing
            history.append(format_error(e))

        steps_taken += 1

    # hit a limit without finishing -- degrade honestly, never fabricate success
    log_incomplete(session_id, trace)
    return "I wasn't able to complete this within my step budget. Here's what I found so far: ..."
Exercise 12.1 -- Design a rollout plan

Plan the launch of an internal agent that drafts customer replies

Write a short rollout plan covering: what percentage of traffic it starts with, what metric decides whether to expand the rollout, and what the fallback behavior is if the agent errors out mid-session.

Show one possible answer

Start at 5% of internal support tickets, agent output shown as a suggested draft that a human must approve before sending (never auto-sent at this stage). Expand rollout only if the eval-set pass rate stays above a set threshold (e.g. 90%) and human edit rate on drafts stays below a set threshold across at least a week of real traffic. Fallback: if the agent errors or times out, the ticket routes to the normal human queue with no visible disruption to the customer.
