REASON PHASE

The LLM as the Reasoning Core

Every agent is only as good as the decisions its underlying model makes. Understanding what the model is actually doing when it 'reasons' will save you from building on top of a misunderstanding.

An LLM does not know facts the way a database does, and it does not plan the way a chess engine does. It predicts plausible continuations of text, one token at a time -- and it turns out that "plausible continuation" is a surprisingly powerful substitute for reasoning, if you set the stage correctly.

What the model can and cannot do reliably

  • Good at: pattern completion, summarizing, translating between formats, following worked examples, breaking a described problem into steps.
  • Unreliable at: precise arithmetic on large numbers, knowing anything after its training cutoff, counting things exactly, guaranteeing factual accuracy without a source to ground it in.
  • The fix in both cases is the same: give the model tools (a calculator, a search function, a clock) rather than asking it to simulate them from memory. This is the entire motivation for agentic design -- offload what the model is bad at to deterministic code, and reserve the model for what it is good at: judgment, language, and flexible reasoning.
Mental model

Think of the LLM as an extremely well-read intern with no long-term memory and no calculator, who is nonetheless excellent at reading a situation and deciding what to do next -- provided you hand them the right references.

For managers

"The model got the number wrong" is usually not a model quality problem -- it is a missing-tool problem. If a task involves exact math, dates, or live data, the fix is to wire up a tool, not to wait for a smarter model.

Two techniques do most of the heavy lifting for getting reliable reasoning out of a model: chain-of-thought and self-consistency.

Chain-of-thought (CoT)

Simply asking a model to "think step by step" before answering measurably improves accuracy on multi-step problems, because it gives the model room to lay out intermediate reasoning as tokens it can then condition on -- rather than jumping straight to a final answer.

# A system prompt fragment that encourages step-by-step reasoning
system_prompt = """
You are a scheduling assistant. Before giving a final answer,
think through the problem step by step inside <thinking> tags,
then give your final answer inside <answer> tags.
"""

Structured output

Agents need the model's output in a predictable shape so code can parse it. Modern APIs support this natively via structured tool-calling schemas (covered in Module 4) rather than asking the model to "return JSON" and hoping.

# Requesting a specific reasoning + answer structure
response = model.call(
    system=system_prompt,
    messages=[{"role": "user", "content": "Can I fit a 45-minute call between my 2pm and 3:15pm meetings?"}]
)
# response.text contains <thinking>...</thinking><answer>...</answer>

Later, when agents chain multiple steps, this reasoning trace becomes invaluable for debugging -- it is the difference between "the agent did something wrong" and "the agent decided X because it believed Y, which was wrong."

Exercise 2.1 -- Spot the failure mode

Diagnose three broken agent behaviors

For each scenario, decide whether the fix is (a) better prompting, (b) adding a tool, or (c) neither is fixable by the model alone:

  1. An agent asked to calculate compound interest over 30 years gives a plausible-looking but slightly wrong number.
  2. An agent asked "what's the weather right now" confidently describes weather from during its training period.
  3. An agent asked to summarize a document jumps straight to a shaky conclusion without showing its work.
Show suggested answers

1. Add a tool -- wire up a calculator or code-execution tool for exact arithmetic.
2. Add a tool -- wire up a live weather API; no amount of prompting fixes a knowledge cutoff.
3. Better prompting -- ask explicitly for step-by-step reasoning before the conclusion (chain-of-thought).
