Tool Use & Function Calling

Tools are how an agent stops talking and starts doing. This is the single most important mechanical skill in the whole course -- everything else assumes you can wire this up correctly.

Function calling is the bridge between a model's decision and a real-world effect. The model never runs code itself -- it outputs a structured request ("call this function with these arguments"), and your application decides whether and how to actually run it.

The four-step tool call cycle

  • Define -- you describe each tool's name, purpose, and expected arguments (usually as a JSON schema) up front.
  • Request -- the model, mid-conversation, outputs a structured request to call a specific tool with specific arguments.
  • Execute -- your code (not the model) actually runs the corresponding function, with whatever validation and permission checks you choose to add.
  • Return -- the result is fed back into the conversation, and the model reasons over it to decide the next step.
This is your security boundary

Every tool you expose is a capability you're granting the model. "Delete file," "send email," and "charge credit card" are all just tools from the model's point of view -- the safety comes entirely from how carefully you scope, validate, and gate each one in your execution layer, not from the model's judgment alone.

For managers

Ask for the tool list of any agent before it ships. It's the clearest, most concrete artifact of what the system can actually do -- more informative than any description of "capabilities."

Here's a minimal, realistic tool-calling setup using the Anthropic API pattern (the shape is very similar across OpenAI, Gemini, and other providers).

# 1. DEFINE -- describe the tool with a schema
tools = [{
    "name": "get_weather",
    "description": "Get the current weather for a given city",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "City name, e.g. 'Ludhiana'"}
        },
        "required": ["city"]
    }
}]

response = client.messages.create(
    model="claude-sonnet-5",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "Should I bring an umbrella in Ludhiana today?"}]
)

# 2. REQUEST -- inspect the response for a tool_use block
for block in response.content:
    if block.type == "tool_use":
        tool_name = block.name          # "get_weather"
        tool_args = block.input         # {"city": "Ludhiana"}

        # 3. EXECUTE -- your code runs the real function, with validation
        result = call_real_weather_api(tool_args["city"])

        # 4. RETURN -- feed the result back for the model to reason over
        follow_up = client.messages.create(
            model="claude-sonnet-5",
            max_tokens=1024,
            tools=tools,
            messages=[
                {"role": "user", "content": "Should I bring an umbrella in Ludhiana today?"},
                {"role": "assistant", "content": response.content},
                {"role": "user", "content": [{
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": str(result)
                }]}
            ]
        )

Designing good tools

  • Narrow and specific beats broad and flexible -- cancel_subscription(user_id) is safer and easier for the model to use correctly than run_database_query(sql).
  • Return errors as data, not exceptions -- let the model see "city not found" and decide what to do, rather than crashing the loop.
  • Put permission checks in the execution layer, never rely on the prompt alone to prevent misuse of a dangerous tool.
Exercise 4.1 -- Design a tool schema

Write the schema for a calendar-booking tool

Design a schedule_meeting tool for an agent that books meetings. Decide: what arguments does it need (attendees, time, duration, title)? What should happen if the requested time conflicts with an existing meeting -- should that be handled inside the tool, or should the tool return conflict data for the model to reason about?

Show one possible answer

Arguments: title (string), attendees (array of emails), start_time (ISO 8601 string), duration_minutes (integer). Best practice: the tool itself should check for conflicts and return structured data like {"booked": false, "conflict": {"title": "1:1 with Sam", "time": "..."}} rather than silently double-booking or silently failing -- this lets the model decide whether to propose an alternate time, ask the user, or escalate.

2. Why is `cancel_subscription(user_id)` generally a safer tool design than `run_database_query(sql)`?

3. If a tool call fails (e.g. city not found), what's generally the best design?
