Multi-Agent Systems & Orchestration

Some tasks are genuinely too broad for one agent to hold in its head at once. Multi-agent design lets you trade a bigger, murkier problem for several smaller, clearer ones -- at the cost of real coordination overhead.

A multi-agent system is not automatically better than a single well-designed agent -- it trades one kind of complexity (a long, complicated single prompt) for another (coordination between agents). Use it when the trade is worth it, not by default.

Common orchestration patterns

  • Orchestrator-worker -- one "manager" agent breaks a goal into sub-tasks and delegates each to a specialized worker agent, then assembles the results. The most common and most controllable pattern.
  • Peer-to-peer / debate -- multiple agents critique or build on each other's outputs before converging on a final answer. Useful for improving quality on subjective or high-stakes outputs.
  • Pipeline -- a fixed sequence of specialized agents, each handing off to the next (research agent -> writer agent -> fact-checker agent). Predictable, easy to debug, less flexible.
When multi-agent actually helps

Splitting into multiple agents earns its overhead when sub-tasks need genuinely different context, tools, or "personas" that would otherwise conflict in one long prompt -- not simply because a task has multiple steps (that's what planning, from Module 6, is for).

For managers

Multi-agent systems multiply your debugging surface and your API cost roughly linearly with the number of agents involved. Before approving one, ask what a single, well-planned agent couldn't do -- if there's no clear answer, you may be paying multi-agent overhead for single-agent value.

Here's a minimal orchestrator-worker pattern -- the safest, most debuggable place to start with multi-agent design.

# Orchestrator decomposes the goal and delegates
def orchestrator(goal):
    subtasks = plan_subtasks(goal)   # e.g. ["research pricing", "research features"]
    results = []
    for subtask in subtasks:
        worker_result = worker_agent(
            role="Research Specialist",
            task=subtask,
            tools=[search_tool]
        )
        results.append(worker_result)
    return synthesize(goal, results)   # orchestrator combines worker outputs

# Each worker is a fully independent agent loop with its own,
# narrower system prompt and its own tool access -- crucially,
# it does not see the full original goal, only its assigned slice.

Coordination failure modes to watch for

  • Context loss -- a worker given too narrow a slice of context makes decisions that don't serve the overall goal.
  • Redundant work -- two workers unknowingly research the same thing because the orchestrator didn't partition the task cleanly.
  • Synthesis failure -- individually correct worker outputs combine into an incoherent or contradictory final result.

Each of these is a real engineering problem, not a hypothetical one -- budget explicit time to test the handoffs between agents, not just each agent in isolation.

Exercise 9.1 -- Design an orchestration for a real task

Break "prepare a competitive analysis report" into an orchestrator-worker design

Decide: how many worker agents, what does each one own, what tools does each need, and what does the orchestrator do with their outputs? Watch out for the redundant-work failure mode above.

Show one possible answer

Orchestrator: takes the list of competitors, assigns one worker per competitor (not per section) to avoid two workers researching the same company. Each worker: a "Competitor Researcher" with a web search tool, given one company name and a fixed set of questions to answer (pricing, key features, recent news). Orchestrator's final step: synthesize all worker outputs into a comparison table and a short narrative summary, explicitly checking for contradictions between workers before finalizing.
