Memory: Short-Term, Long-Term & Vector Stores

An agent without memory relearns everything every single turn. Getting memory right is what separates a one-off script from an assistant that actually gets better the more it's used.

Every LLM call is stateless -- the model remembers nothing between calls except what you explicitly put back into its context window. "Memory" in an agent is really just an engineering pattern for deciding what to re-inject, and when.

Three kinds of memory agents actually use

  • Working memory -- the current conversation history, kept in the context window turn by turn. Cheap, but limited by context size and gets expensive at scale.
  • Episodic / long-term memory -- facts and events from past sessions, stored externally (often in a vector database) and retrieved only when relevant to the current turn.
  • Procedural memory -- learned preferences or corrections ("always format dates as DD/MM/YYYY for this user") that get folded back into future system prompts rather than raw conversation history.
Key idea: retrieval, not recall

Long-term memory in an agent isn't the model "remembering" -- it's your system searching a store of past information for what's relevant right now, and injecting only that into the prompt. This is exactly the same mechanism as RAG (Module 7), just applied to the agent's own history instead of external documents.

For managers

"Does it remember me?" is really a data-retention and privacy question as much as a technical one. Decide upfront what gets stored, for how long, and how a user can see or delete it -- this is far easier to design in than to retrofit.

A practical long-term memory system has three moving parts: an embedding model to turn text into vectors, a vector store to hold them, and a retrieval step that runs before every agent turn.

# Writing to long-term memory after a conversation
summary = summarize_for_memory(conversation_transcript)
embedding = embed_model.encode(summary)
vector_store.upsert(
    id=f"user:{user_id}:{timestamp}",
    vector=embedding,
    metadata={"user_id": user_id, "text": summary, "date": timestamp}
)

# Retrieving relevant memories before responding
query_embedding = embed_model.encode(current_user_message)
relevant_memories = vector_store.query(
    vector=query_embedding,
    filter={"user_id": user_id},
    top_k=3
)

# Injected into the system prompt, not the raw conversation history
system_prompt = f"""
Relevant things you know about this user from past conversations:
{format_memories(relevant_memories)}
"""

Context window management

Even working memory needs a strategy once conversations get long. Common approaches: sliding window (keep the last N turns), rolling summarization (periodically compress older turns into a summary), or hybrid (keep recent turns verbatim, summarize everything older).

Exercise 5.1 -- Design a memory policy

Decide what a customer-service agent should and shouldn't remember

You're designing memory for an agent that helps customers troubleshoot a product across multiple sessions. List: (a) two things worth storing in long-term memory, (b) one thing that should not be stored for privacy reasons, and (c) how long memories should be retained.

Show one possible answer

(a) Worth storing: the product model/version the customer owns (avoids re-asking every session), and a summary of unresolved issues so a new session picks up where the last left off.
(b) Should not store: payment details or anything typed that looks like a password or verification code, even if mentioned in passing.
(c) A reasonable default is a rolling retention window (e.g. 12 months) with a clear, user-facing way to view and delete stored memories on request.
